# Shadow Testing

## 1. Definition & Core Concepts
Shadow Testing (also called mirrored testing) is the practice of routing a copy of live production traffic to a new AI model version or prompt configuration in the background, while returning only the baseline version's response to the user. This allows teams to validate execution behavior under realistic conditions without affecting the user experience.

## 2. Why It Exists / What Problem It Solves
Canary deployments still expose a fraction of users to potential errors. Shadow testing provides a zero-risk mechanism to verify new models or prompts. By mirroring 100% of live traffic to the shadow version, teams can evaluate latency, API cost, resource usage, formatting compliance, and generation quality at full production scale.

## 3. What Breaks in Production Without It
- **API Cost Overruns:** Deploying a new prompt version that is unexpectedly verbose or causes infinite tool call loops, triggering massive bills.
- **Latency regressions:** Moving to a new self-hosted model that cannot handle concurrent production loads, causing gpu memory crashes.
- **Silent Parser Errors:** Releasing a new model that subtly alters output format structure, breaking downstream parsing scripts.

## 4. Best Practices
- **Implement Asynchronous Mirroring:** Use message queues (like Kafka) or non-blocking async loops to clone requests, dispatching them to the shadow model without delaying the primary user response.
- **Discard Shadow Outputs to Users:** Ensure that client interfaces never see or parse the outputs generated by the shadow deployment.
- **Perform Automated Side-by-Side Evaluations:** Run automated LLM-as-a-judge comparison tests on baseline vs. shadow outputs to flag variations in response quality, tone, or safety.

## 5. Common Mistakes / Anti-Patterns
- **Executing Write-enabled Tools in Shadow:** Allowing a shadow agent to call tools that modify databases, charge credit cards, or send emails, leading to duplicate actions.
- **Blocking the Main Path:** Executing the shadow request synchronously on the user thread, causing response latencies to double.

## 6. Security Considerations
- **Data Protection Compliance:** Ensure that user inputs containing sensitive PII are handled in shadow testing with the same strict compliance controls (e.g. data isolation, encryption) as the baseline path.

## 7. Performance Considerations
- **GPU Resource Planning:** Shadow testing doubles the query load. Ensure that your GPU clusters or model provider quotas have sufficient capacity to handle twice the typical query volume.

## 8. Scalability Considerations
- **Decoupled Analytics Pipeline:** Stream shadow results and comparisons to an offline data warehouse (e.g. BigQuery, Snowflake) rather than writing them to production transactional stores.

## 9. How Major Companies Implement It
- **Stripe:** Routes a duplicate stream of all user prompts to shadow endpoints when evaluating new proprietary model updates, verifying that latency and semantic correctness meet requirements before switching.

## 10. Decision Checklist (Shadow Testing Suitability)
- Use **Shadow Testing** when:
  - Evaluating major model migrations or complex agent refactors where quality regressions could cause critical business damage.
- Skip **Shadow Testing (use Canary/A/B instead)** when:
  - Working on write-heavy workflows, transaction executions, or interactive multi-step dialogues that cannot be easily simulated in isolation.

## 11. AI Coding-Agent Guidelines
- Write middleware that clones query payloads and pushes them to async worker threads for shadow model execution, ensuring write-enabled tools are mocked.

## 12. Reusable Checklist
- [ ] Asynchronous request mirroring implemented (non-blocking)
- [ ] Shadow outputs isolated and blocked from user delivery
- [ ] Write-enabled tools disabled or mocked in the shadow environment
- [ ] GPU clusters or API limits sized to support double traffic load
- [ ] Automated side-by-side comparison (LLM-as-a-judge) active
- [ ] Shadow latency, error rates, and API costs monitored separately
