# Deterministic vs. AI Features

## 1. Definition & Core Concepts
Deterministic features are software modules governed by strict, rule-based algorithms where the same input always produces the exact same output. AI features are non-deterministic modules governed by probabilistic models where the same input can produce varying outputs.

## 2. Why It Exists / What Problem It Solves
It ensures system architects do not swap out stable, reliable code algorithms for unstable, expensive generative models. Fulfilling requirements with deterministic code guarantees system reliability and keeps compute costs low.

## 3. What Breaks in Production Without It
- **Calculation Inconsistencies:** Billing calculators output varying monthly prices due to LLM reasoning drift.
- **Unpredictable API responses:** JSON payload formats break because the model generated a text summary instead of a matching schema.
- **State Machine Deadlocks:** Workflow states get stuck because the model output an un-mapped custom state string.

## 4. Best Practices
- **Deterministic Core, AI Edge:** Keep core ledger, state machine, and data validations deterministic; restrict AI to formatting, searching, or summarization.
- **Strict Parsing Schemes:** Enforce JSON schemas on AI outputs, using deterministic code to validate the results.
- **Accuracy Fallbacks:** Implement heuristic backups to catch model translation failures.

## 5. Common Mistakes / Anti-Patterns
- **LLMs for Calculations:** Using an LLM to calculate invoice tax rates or cart totals.
- **LLMs for State Transitions:** Letting an LLM decide database state shifts directly without validating inputs against a strict state machine schema.

## 6. Security Considerations
- **Output Sanitization:** Deterministic validators must sanitize all text outputs generated by AI to prevent cross-site scripting (XSS) or SQL injection vectors.

## 7. Performance Considerations
- **Throughput Sizing:** Deterministic algorithms execute in microseconds ($<1\text{ms}$); LLM calls take seconds ($>1000\text{ms}$). Keep all critical path steps deterministic.

## 8. Scalability Considerations
- **Cost Scaling:** Deterministic calculations scale linearly with negligible cost; AI scaling requires monitoring model token usage.

## 9. How Major Companies Implement It
- **Stripe:** Uses deterministic rule engines to process card transactions and fraud limits; LLMs are restricted to support documentation answers.
- **Amazon:** Uses deterministic databases to manage product inventory; LLMs are used to summarize customer product reviews.

## 10. Decision Checklist (Technology Mapping)
- Use **Deterministic Code** when:
  - Performing arithmetic, tax, or ledger calculations.
  - Validating user login credentials or checking security tokens.
  - Transitioning database order status values.
- Use **AI Features** when:
  - Translating language texts or summarizing customer chat logs.
  - Doing semantic clustering or vector comparisons.

## 11. AI Coding-Agent Guidelines
- Never replace an existing database validation or business logic check with an LLM call. Keep the core architecture deterministic.

## 12. Reusable Checklist
- [ ] Core business ledger equations implemented using deterministic code
- [ ] Database state machine transitions validated using strict enums
- [ ] AI output strings parsed and validated against JSON schemas in code
- [ ] General sanitization filters active on all AI-generated outputs
- [ ] Latency budget confirmed for any route containing an LLM call
- [ ] Fallback checks implemented to catch malformed AI outputs
