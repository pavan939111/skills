# Human-in-the-Loop (HITL)

## 1. Definition & Core Concepts
Human-in-the-Loop (HITL) is the design pattern where human verification, approval, or correction steps are integrated into an AI workflow to review outputs before actions execute in production.

## 2. Why It Exists / What Problem It Solves
AI models are probabilistic and hallucinate. For critical actions (e.g. sending bulk marketing campaigns, applying discounts, deleting users), HITL acts as a safety barrier, preventing model errors from directly affecting users or databases.

## 3. What Breaks in Production Without It
- **Runaway Email Spams:** Customer outreach agents generate malformed or offensive texts and send them to thousands of clients automatically.
- **Financial Leakages:** AI refund agents process unverified return claims, paying out cash to fraud rings.
- **Accidental State Deletions:** System administration agents run cleanups that delete active customer records based on misclassified instructions.

## 4. Best Practices
- **Implement Approval States:** Save AI outputs as a "pending_approval" database state; require human admin authorization to promote to "executed".
- **Design Review UIs:** Provide comparison screens displaying original input, AI draft, and edit options.
- **Set Risk Escalation Limits:** Automate review paths based on transaction limits (e.g., refund under $20 = auto, over $20 = HITL).

## 5. Common Mistakes / Anti-Patterns
- **Direct SMTP integration:** Letting AI text generation models send emails directly without holding drafts in database queues.
- **Bypassing validation checks:** Assuming human-reviewed AI text is safe, ignoring standard database character sanitizations.

## 6. Security Considerations
- **Credential Separation:** Reviewers must execute actions using their own session tokens, preventing agents from calling write APIs directly.

## 7. Performance Considerations
- **Asynchronous Task Sizing:** HITL steps add delays. Move approval actions into background job queues (e.g., SQS + database workers) to prevent web thread locks.

## 8. Scalability Considerations
- **Reviewer Queues:** Implement load balancing across admin dashboard screens to distribute review workloads.

## 9. How Major Companies Implement It
- **Grammarly:** Presents grammar suggestions to the user for explicit click-to-accept actions (never mutates document source code directly).
- **Stripe:** Uses AI to identify fraudulent charge patterns, adding cases to analyst review dashboards instead of auto-cancelling accounts.

## 10. Decision Checklist (HITL Requirements)
- Use **Fully Automated (No HITL)** when:
  - Task is read-only search indexing or internal summarization.
  - Error impact does not alter database state or client interfaces.
- Use **Human-in-the-Loop (HITL)** when:
  - Task performs database mutations, sends emails, or processes payments.
  - Transaction values exceed configured thresholds.

## 11. AI Coding-Agent Guidelines
- Always implement state transitions (`Draft` $\rightarrow$ `Pending Review` $\rightarrow$ `Approved`) for any write actions generated by LLM models.

## 12. Reusable Checklist
- [ ] Database approval states defined (`pending_approval`, `approved`, `rejected`)
- [ ] Direct network API write actions disabled for autonomous agent loops
- [ ] Action risk thresholds set (e.g. transaction values trigger HITL)
- [ ] Review dashboard UI designed with comparison and editing views
- [ ] Automated notification hooks set up to alert reviewers of backlogs
- [ ] Admin edit actions logged with audit trails
