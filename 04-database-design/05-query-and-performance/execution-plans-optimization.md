# Execution Plans

## 1. Definition & Core Concepts

An Execution Plan is the step-by-step sequence of physical operations generated by the database query optimizer to retrieve or modify data for a given SQL statement.

Core plan concepts:
- **`EXPLAIN`:** The SQL prefix command that displays the query planner's estimated execution steps, cost values, and row sizes without actually executing the query.
- **`EXPLAIN ANALYZE`:** The SQL command that actually executes the query, returning the real execution times, actual row counts, and buffer usage alongside the estimates.
- **Scan Operations:**
  - *Seq Scan (Sequential Scan):* Reading the entire table from disk.
  - *Index Scan:* Traversing the B-Tree index to locate matching keys, then reading data pages from the heap.
  - *Index Only Scan:* Reading data directly from the index tree (covering index) without touching the table heap.
  - *Bitmap Index Scan:* Reading index pointers into an in-memory bitmap array, sorting by disk location, and retrieving data blocks, optimizing random disk I/O.
- **Join Operators:**
  - *Nested Loop:* Iterating through the outer table and executing index lookups on the inner table for every row. (Efficient for small datasets).
  - *Hash Join:* Creating an in-memory hash table of the smaller table, then scanning the larger table to find matches. (Efficient for unsorted, medium-to-large datasets).
  - *Merge Join:* Sorting both datasets by the join key and merging them. (Efficient for very large, pre-sorted datasets).

*(Boundary Note: Code-level query profiling wrappers, transaction log dashboards, and connection trace configurations belong in `backend-development/`. This document covers SQL explain syntax, scan/join node types, buffer tracking, and plan analysis.)*

## 2. Why It Exists / What Problem It Solves

SQL is a declarative language; you write *what* data you want, not *how* to retrieve it. The database engine calculates multiple ways to execute the query and generates an execution plan. Reviewing this plan is the only way to diagnose query slowness, showing if the engine is performing slow sequential scans on disk or using index seeks.

## 3. What Breaks in Production Without It

- **Slow Query Outages from Plan Drift:** Stale database statistics cause the optimizer to choose a slow sequential table scan instead of an index seek on a table containing 100M rows, causing API timeouts.
- **Production State Mutations via EXPLAIN ANALYZE:** Running `EXPLAIN ANALYZE UPDATE user SET is_active = false` in a production database to profile write latency. Unlike `EXPLAIN`, `EXPLAIN ANALYZE` actually executes the update, modifying production data.
- **Disk Saturation from In-Memory Sort Spills:** A join operation runs out of query work memory (e.g. `work_mem`), forcing the engine to spill sorting tasks to temporary disk files, degrading performance.
- **Testing Blindness on Empty Databases:** Running `EXPLAIN` in local development containing only 10 rows of mock data. Relational optimizers will always choose sequential scans for small tables, making it impossible to verify if indexes will be used at production scale.

## 4. Best Practices

- **Profile with EXPLAIN ANALYZE Safely:** Always run `EXPLAIN (ANALYZE, BUFFERS)` on slow queries in staging environments containing production-scale data to capture execution times and buffer reads.
- **Inspect Estimated vs. Actual Row Counts:** Compare the query planner's `rows` estimate to the `actual rows` returned by the scan nodes. If they differ by orders of magnitude (e.g., estimating 1 row but returning 1,000,000), database statistics are stale. Run `ANALYZE` on the table.
- **Scan for Sequential Scans (Seq Scan):** Look for `Seq Scan` nodes in tables containing over 10,000 rows. Investigate if adding an index or rewriting the query filters can convert the scan to an `Index Only Scan` or `Index Scan`.
- **Verify Buffer Hits:** Look for the `Buffers:` output in PostgreSQL plans:
  - *Shared Hit:* Pages read from database RAM memory.
  - *Shared Read:* Pages read from disk.
  Target queries that keep shared reads minimal and shared hits high.
- **Always Wrap Mutations in Rollbacks when Profiling:** If running `EXPLAIN ANALYZE` on a write mutation (INSERT/UPDATE/DELETE) to profile write paths, wrap the command in a transaction block and roll it back:
  `BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;`

## 5. Common Mistakes / Anti-Patterns

- **EXPLAIN ANALYZE on Live Production Writes:** Executing write mutations directly with `ANALYZE` flags in production.
- **Analyzing without Representative Data:** Profiling query plans on empty local databases.
- **Ignoring Cost Indicators:** Focusing solely on execution times without analyzing the node cost distributions to identify which sub-query node is the bottleneck.
- **Allowing Hash Join Spill to Disk:** Ignoring plans that show hash joins exceeding memory limits (e.g., `spill to disk`), slowing down performance.

## 6. Security Considerations

- **Redacting Parameter Values:** Execution plans print actual parameter values used in query filters. Ensure execution plan logging tools redact sensitive customer data (like passwords, keys, PII) before writing to monitoring files.

## 7. Performance Considerations

- **Optimizer Overhead:** The database optimizer consumes CPU to generate execution plans. Avoid writing SQL statements containing dozens of joins, as the compilation cost of calculating the plan can exceed the query execution time itself.

## 8. Scalability Considerations

- **Plan Drift Monitoring:** Execution plans can change over time as tables grow and database statistics update. Continuously monitor query plan patterns using tools like pg_store_plans to detect and fix plan drift.

## 9. How Major Companies Implement It

- **Stripe:** Automatically attaches PostgreSQL `EXPLAIN` execution plans to query monitoring dashboards, allowing developers to analyze query execution paths when a database query exceeds defined latency limits.
- **Netflix:** Uses automated execution plan analyzers in staging pipelines, blocking deployments if changes introduce table scans on transactional tables.

## 10. Decision Checklist (Execution Node Diagnostics)

Audit execution plans for the following node warnings:

- Replace **Seq Scan (Large Table)** when:
  - The table has over 10,000 rows and the query filters on unindexed columns (add B-Tree index).
- Run **ANALYZE table** when:
  - Estimated rows differs from actual rows by more than 10x (stale statistics).
- Increase **work_mem (Sort Memory)** when:
  - Sort or Hash Join nodes show `spilled to disk` or `temp read/write` (allocate more query sort memory).
- Wrap **EXPLAIN ANALYZE** in transactions when:
  - Profiling write queries (INSERT, UPDATE, DELETE).

## 11. AI Coding-Agent Implementation Guidelines

- Always check query execution plans on staging-scale datasets before recommending schema indexes.
- Never execute `EXPLAIN ANALYZE` on write mutations in production without wrapping them in transaction rollbacks.
- Always check that estimated row counts align with actual row counts in plan outputs.
- Never write SQL queries with excessive joins (>10 tables) that overload the database optimizer's plan compilation phase.
- Always target converting `Seq Scan` nodes to `Index Only Scan` or `Index Scan` nodes.

## 12. Reusable Checklist

- [ ] Query plans analyzed using `EXPLAIN` or `EXPLAIN ANALYZE` on representative datasets
- [ ] No sequential scans (`Seq Scan`) on large tables (>10,000 rows)
- [ ] Estimated row counts match actual row counts in execution logs (no stale statistics)
- [ ] Sort and Hash operations execute in RAM (no temp file spills to disk)
- [ ] `EXPLAIN ANALYZE` on write mutations wrapped in `BEGIN` and `ROLLBACK` blocks
- [ ] Database statistics updated via `ANALYZE` if plan estimates are skewed
- [ ] Joins execute using optimal operators (Nested Loop for small, Hash/Merge for large datasets)
- [ ] Sensitive parameter values redacted from execution plan log files
