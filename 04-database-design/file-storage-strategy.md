# File Storage

## 1. Definition & Core Concepts

File storage at the application level involves the management, ingestion, and delivery of unstructured binary data (e.g., images, video uploads, PDF invoices, CSV exports).

Core pieces:
- **Object Storage:** Flat, key-value storage designed for large files (e.g., Amazon S3, Google Cloud Storage, Azure Blob Storage). Unlike hierarchical file systems, it scales infinitely.
- **Local Ephemeral Storage:** The filesystem inside a container. It is lost when the container stops or restarts.
- **Pre-signed URLs:** Temporary, cryptographically signed URLs generated by the application that grant direct read or write access to a specific object in storage, bypassing the application servers.
- **Multipart Upload:** Splitting a large file into smaller chunks and uploading them in parallel, reducing the impact of network drops.
- **Content Delivery Network (CDN):** A geographically distributed network of proxy servers that cache and serve static files closer to users (e.g., Cloudflare, AWS CloudFront).

*(Boundary Note: Database-specific storage formats like DB BLOBs are out of scope. This document focuses on code-level integration with file-handling systems, security sanitization, and storage abstractions.)*

## 2. Why It Exists

Application servers are designed to be stateless and ephemeral so they can scale horizontally. Storing user uploads on a local server's hard drive violates statelessness, limits storage capacity, and results in data loss during container redeployments. Offloading files to dedicated object storage ensures durability, availability, and low-cost scalability.

## 3. What Breaks in Production Without It

- **Data Loss on Redeployment:** Files uploaded to local server directories vanish the moment the cloud platform autoscales, crashes, or updates the server image.
- **Out of Memory Crashes:** Code that reads an entire 1GB uploaded file into RAM (`fs.readFileSync`) crashes the application process instantly due to memory exhaustion.
- **Path Traversal Security Vulnerabilities:** Trusting user-supplied filenames (e.g., uploading a file named `../../etc/passwd`) allows attackers to overwrite critical system files.
- **Server Disk Exhaustion:** Temp files left behind by export jobs fill up the server's root disk space, crashing the operating system.
- **Server Thread Starvation:** Routing large file downloads directly through application threads blocks event loops and exhausts worker pools, causing timeout failures for API clients.

## 4. Best Practices

- **Use the Storage Adapter Pattern:** Write file operations behind an interface/adapter. This allows running a local filesystem adapter for unit tests, while swapping in an S3/GCS adapter in production without changing business logic.
- **Generate Random Unique Filenames:** Never trust the user's filename. Generate a unique key (e.g., UUID/v4) on the server, append the validated file extension, and store the original filename as database metadata if needed.
- **Offload Uploads and Downloads via Pre-signed URLs:** Do not funnel file bytes through application servers. Generate a write-enabled pre-signed URL so the client upload goes directly to S3. Generate a read-enabled pre-signed URL for protected files to offload downloading.
- **Stream Large Files:** When processing files locally (e.g., parsing a CSV), always use streaming APIs (e.g., node streams, python generators). Read and process data in chunks to keep memory usage low and constant.
- **Configure Lifecycle Rules:** Set automatic expiration policies in object storage (e.g., auto-delete files in `tmp/exports/` after 24 hours) to control storage costs.
- **Strip Metadata on Upload:** Use libraries to strip EXIF metadata (GPS coordinates, camera models, creation dates) from user-uploaded images to protect privacy.

## 5. Common Mistakes / Anti-Patterns

- **Storing Files in DB BLOB Columns:** Storing image/file bytes directly in relational databases, which inflates database backups and degrades search performance.
- **Hardcoding Local Paths:** Referencing absolute paths like `C:\uploads\` or `/var/www/uploads` which fail on different developer machines or container environments.
- **Serving Static Assets Through Application Code:** Using Node/Python to serve images or CSS files, rather than offloading them to a CDN or static web server (NGINX/S3).
- **Publicly Open Buckets:** Configuring object storage buckets to be public by default, leading to accidental leaks of corporate or personal documents.

## 6. Security Considerations

- **Malicious File Execution:** Attackers uploading executable scripts (e.g., `.php`, `.sh`, `.exe`) and attempting to execute them on the server. Disable execution permissions on upload directories and restrict accepted file extensions via mime-type verification (not just filename checking).
- **Server-Side Request Forgery (SSRF):** If the application downloads a file from a user-supplied URL, validate that the URL is not pointing to internal network IP addresses (like `169.254.169.254` or `localhost`).

## 7. Performance Considerations

- **Cache-Control Headers:** Always set far-future `Cache-Control` headers (e.g., `public, max-age=31536000, immutable`) for public files served via CDNs, ensuring browsers do not keep asking servers for unchanged files.
- **Image Compression:** Compress and resize images on upload to save network bandwidth and storage.

## 8. Scalability Considerations

- **CDN Edge Isolation:** Never point users directly to object storage domains for public assets. Route static asset requests through a CDN domain to prevent hitting object storage request rate limits.

## 9. How Major Companies Implement It

- **Slack:** Uses AWS S3 for all user file shares. They generate short-lived, encrypted pre-signed URLs dynamically on the backend to authenticate file downloads, ensuring files are only visible to authorized team members.
- **Netflix:** Encodes video chunks in parallel and stores final video streams on AWS S3. These files are pushed to thousands of Open Connect CDN appliance boxes deployed directly inside internet service provider (ISP) networks globally.

## 10. Decision Checklist

- Use **Object Storage (S3/GCS)** when: The app is horizontally scaled, files are persistent, file sizes are large, or files must be accessed securely by clients.
- Use **Local Temporary Storage (`/tmp`)** ONLY when: Creating temporary, ephemeral scratchpad files that will be deleted within the lifecycle of the request/job.

## 11. AI Coding-Agent Implementation Guidelines

- Always hide file storage implementations behind a clean adapter interface (e.g., `StorageService`).
- Never use raw file paths from user inputs — always sanitize paths and isolate directories, or generate unique UUID keys.
- Always use stream/buffer chunking interfaces when importing or exporting files larger than 10MB.
- Never write credentials, key files, or S3 secrets in the codebase — load them via IAM roles or environment settings.
- Always verify MIME types of uploaded files using magic number checks/content headers, not just string splits on the file extension.
- Always configure upload configurations to limit maximum request body size (e.g., reject files > 50MB before parsing).

## 12. Reusable Checklist

- [ ] File operations hidden behind a mockable Storage Adapter interface
- [ ] EPHEMERAL warning: No persistent files stored on the local container disk
- [ ] User filenames replaced with generated UUID keys before saving
- [ ] File uploads and downloads offloaded to pre-signed URLs where possible
- [ ] Large file processing uses streams/chunking; no `readFileSync` of big payloads
- [ ] Uploaded image EXIF metadata stripped for privacy
- [ ] Maximum upload file size limits enforced at the boundary
- [ ] File access permissions set strictly (private by default, pre-signed for read access)
