# Vector Database

## 1. Definition & Core Concepts

A Vector Database is a database designed to store, index, and query high-dimensional mathematical vectors (embeddings) generated by Machine Learning models (e.g. LLMs, image encoders).

Core pieces:
- **Vector Embeddings:** Numerical arrays representing the semantic meaning of unstructured data (text, images, audio).
- **Distance Metrics:** Mathematical formulas used to calculate similarity between vectors:
  - *Cosine Similarity:* Measures the angle between vectors (values 1 to -1, standard for text search).
  - *Dot Product:* Measures angle and magnitude (standard for normalized embeddings).
  - *Euclidean Distance (L2):* Measures the straight-line distance between points.
- **ANN (Approximate Nearest Neighbor) Search:** Indexing algorithms that search high-dimensional space rapidly, trade-off complete precision for sub-millisecond search speeds (e.g., HNSW - Hierarchical Navigable Small World, IVF - Inverted File Index).
- **Metadata Filtering:** The ability to filter search results using standard relational keys (e.g., `tenant_id`, `created_at`) alongside vector calculations.

*(Boundary Note: Machine learning model execution, embedding generation APIs (like OpenAI client calls), and LLM RAG pipelines belong in `backend-development/`. This document covers vector-database engine selection, indexing configurations, and similarity math parameters.)*

## 2. Why It Exists / What Problem It Solves

Unstructured data (like search queries, images, or customer service tickets) cannot be compared using standard SQL text matching or B-Tree indexes. Relational databases cannot calculate multi-dimensional distances (e.g. comparing 1536-dimensional vectors) in real-time, requiring full table scans. Vector databases index these dimensions, enabling semantic search ("find documents similar to X in meaning") across millions of records in milliseconds.

## 3. What Breaks in Production Without It (or When Misapplied)

- **Search Latency Spikes from RAM Starvation:** The HNSW indexing algorithm requires storing the entire vector graph structure in RAM. If the database node runs out of memory, it swaps to disk, increasing search times from 5ms to 5,000ms.
- **Semantic Drift (Incorrect Distance Metrics):** Choosing a distance metric that does not match the embedding model's training config (e.g. using Euclidean distance on models trained for Cosine similarity). The database returns irrelevant search results.
- **Severe Performance Lag from Post-Filtering:** The database performs vector similarity search first, retrieves 10 items, and then filters out items that do not match a metadata tag (like `tenant_id`), returning empty or incorrect results. Pre-filtering indexes are required.
- **Index Build Bottlenecks:** Inserting vectors in high-volume loops locks index reconstruction pipelines, blocking concurrent search queries.

## 4. Best Practices

- **Match Distance Metrics to the Model:** Always configure the database index to use the exact distance metric (Cosine, Dot Product, or L2) that the embedding model was trained on.
- **Enforce Metadata Pre-Filtering:** Configure indices to evaluate metadata constraints *before* performing vector distance calculations, ensuring results match security boundaries (e.g., filtering by `tenant_id` first).
- **Size Server RAM for HNSW Indices:** Calculate memory requirements beforehand. For HNSW indexes, allocate enough RAM to hold: $(Number\ of\ Vectors \times Vector\ Dimensions \times 4\ bytes) \times Overhead\ Multiplier\ (typically\ 1.5\ to\ 2.0)$.
- **Tune ANN Parameters for Recall Accuracy:** Balance speed and precision:
  - Increase `efSearch` and `efConstruction` parameters in HNSW configurations to improve search recall accuracy.
  - Decrease them to reduce query latency.
- **Keep Raw Payloads Lightweight:** Do not store large text blocks or files inside the vector database records. Store the vector, low-cardinality metadata tags, and the primary key ID of the document, referencing the actual text in a primary relational database.

## 5. Common Mistakes / Anti-Patterns

- **Storing Large Document Texts:** Inlining 10MB document texts inside vector fields, exhausting database storage and memory bandwidth.
- **Mismatched Embedding Models:** Storing vectors generated by different models (e.g. mixing 512-dimension vectors with 1536-dimension vectors) in the same index.
- **Querying Un-embedded Text:** Passing raw string searches to the vector database expecting semantic parsing (text must always be converted to embeddings via client APIs before querying).
- **Using Vector DB as Primary Relational DB:** Attempting to run standard CRUD operations, joins, or relational updates in a vector-only store.

## 6. Security Considerations

- **Strict Tenant Isolation Gating:** Vector graphs naturally link nodes. Ensure metadata pre-filters are configured to validate tenant boundaries strictly on every search query to prevent data leaks.

## 7. Performance Considerations

- **Index Rebuild Latency:** Creating an ANN index (especially HNSW) is CPU and memory intensive. Schedule bulk inserts during off-peak hours, or use asynchronous indexing configurations if supported.

## 8. Scalability Considerations

- **Horizontal Sharding of Vector Space:** Partition vector indexes horizontally based on metadata dimensions (e.g. sharding by tenant or creation year) to restrict vector space search limits on individual nodes.

## 9. How Major Companies Implement It

- **Shopify:** Utilizes vector search databases to power semantic product discovery, allowing users to search descriptions (e.g., "warm winter clothes") and receive relevant matches without relying on exact keyword inputs.
- **Netflix:** Stores vector representations of user profiles and movie features in distributed caches, performing real-time similarity calculations to feed recommendation models.

## 10. Decision Checklist (when to use / when NOT to use)

- Use **Vector Databases (Pinecone, Qdrant, Milvus, pgvector)** when:
  - Building semantic search engines, recommendation engines, or image-matching tools.
  - Feeding context data to LLM applications (Retrieval-Augmented Generation - RAG).
  - Queries compare mathematical distance similarity of high-dimensional embeddings.
- Use **Relational or Search Engine Databases** when:
  - Queries rely on exact keyword match, prefix wildcards, or structured conditions.
  - Traditional transactional constraints (ACID) are required.
  - Data consists of flat, structured records with no vector representation.

## 11. AI Coding-Agent Implementation Guidelines

- Always set the database distance metric to match the embedding provider's specification (e.g., Cosine for OpenAI text-embedding-ada-002).
- Never store large text payloads inside vector database rows — reference document primary keys.
- Always implement metadata filters (like `tenant_id` check) in the vector search query body.
- Never write search queries that omit similarity threshold limits (e.g., return matches with similarity score > 0.7).
- Always verify that target database server RAM exceeds HNSW vector storage formulas.

## 12. Reusable Checklist

- [ ] Distance metric matches the embedding model training specifications
- [ ] Database server RAM sized to hold the entire HNSW index in memory
- [ ] Metadata queries evaluate via pre-filtering (filtering before vector calculations)
- [ ] Vector records store reference IDs, not large raw text files
- [ ] All vectors in an index have matching dimensions (no model mixing)
- [ ] Access control filters (e.g., `tenant_id`) enforced on every query
- [ ] Similarity score thresholds configured to filter out low-match noise
- [ ] Bulk indexing strategies configured for initial vector loading
