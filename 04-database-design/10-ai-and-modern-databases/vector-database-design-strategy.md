# Vector Database Design

## 1. Definition & Core Concepts

Vector Database Design is the discipline of structuring, indexing, and querying high-dimensional vector embeddings (mathematical representations of data meaning generated by AI models) to perform efficient semantic similarity searches.

Core concepts:
- **Vector Embeddings:** Arrays of floating-point numbers (typically 256 to 1536 dimensions) representing the semantic features of text, images, or audio.
- **ANN (Approximate Nearest Neighbor) Search:** Algorithms that search high-dimensional space to find the closest vectors to a query, trading 100% exact accuracy for search speed ($O(\log N)$ instead of $O(N)$).
- **HNSW (Hierarchical Navigable Small World):** A graph-based ANN index structure that links vectors in multi-layer graphs, enabling fast, high-recall semantic searches. (High memory footprint).
- **IVF (Inverted File Index):** A cluster-based ANN index that partitions vector space into Voronoi cells, scanning only the closest cluster centroids. (Lower memory footprint, slower search).
- **Distance Metrics:** Mathematical formulas used to calculate vector similarity:
  - *Cosine Similarity:* Measures the angle between vectors (ignores magnitude). Standard for text matching.
  - *Dot Product:* Measures directional alignment and magnitude. Fastest to compute if vectors are normalized.
  - *Euclidean (L2) Distance:* Measures the straight-line distance between points in space.

## 2. Why It Exists / What Problem It Solves

Traditional databases search using exact matches (SQL `LIKE` or index scans on B-Trees). However, exact matches cannot understand meaning (e.g. searching for "automobile" misses rows containing "car"). Vector databases store embedding vectors in multi-dimensional space. By executing similarity distance calculations (ANN search), they find semantically related content, enabling semantic search, recommendation engines, and Retrieval-Augmented Generation (RAG) contexts.

## 3. What Breaks in Production Without It

- **Out of Memory (OOM) Crashes from Graph Bloat:** Sizing RAM based on raw vector float arrays without calculating the HNSW graph overhead. HNSW requires substantial extra RAM (often 1.5x to 2x the raw vector size) to store graph nodes, causing host servers to crash.
- **Unusable Search Latency (Table Scans):** Querying vectors without defining database vector indexes. The engine must compute the exact distance (Flat search) for every single row in the table, spiking CPU and query times past seconds.
- **Poor Search Recall from Untuned HNSW:** Accepting default index parameters under large datasets. The search misses highly relevant context documents (low recall) because the graph connections are too sparse.
- **Index Build Lockouts:** Running index generation synchronously on massive datasets, freezing write paths and locking database CPU at 100% for hours.

## 4. Best Practices

- **Match Distance Metrics to the Embedding Model:** Always select the exact distance metric used during the training of the embedding model:
  - Use *Cosine Similarity* for standard text models (e.g., OpenAI text-embedding-3-small).
  - Use *Dot Product* for normalized embeddings to optimize query execution speeds.
- **Enforce Index Sizing (Keep in RAM):** Ensure the entire HNSW index fits within the server's RAM buffer. If the index spills to disk, query latency collapses due to slow random I/O.
- **Tune HNSW Parameters for Recall and Speed:**
  - *M:* Max number of bidirectional links per node (e.g. `M = 16` for standard text, `M = 64` for complex image searches).
  - *efConstruction:* Search depth during index build (e.g. `efConstruction = 64` to `200`). Higher values increase build time but improve recall.
- **Apply Quantization (PQ/SQ) to Compress Vectors:** If RAM is limited, configure Scalar Quantization (SQ8) or Product Quantization (PQ) to compress 32-bit floats to 8-bit integers, reducing memory footprint by up to 75% at the cost of minor recall loss.
- **Build Vector Indexes Asynchronously:** Generate vector indexes asynchronously in the background, or load data in bulk before applying the index to prevent write path locks.

## 5. Common Mistakes / Anti-Patterns

- **Euclidean Distance for Normalized Text:** Using Euclidean (L2) distance for model embeddings designed for Cosine metric, leading to poor semantic matches.
- **Vector Search on B-Trees:** Attempting to query vectors using B-Tree indexes, which are designed for 1D scalar data, not multi-dimensional arrays.
- **Ignoring HNSW Memory Overhead:** Under-provisioning RAM, causing OOM crashes.
- **Indexing Low-Cardinality Vectors:** Creating vector indexes on low-dimensional or non-semantic lists, wasting resources.

## 6. Security Considerations

- **Vector Reconstruction Attacks:** Attackers with read-only access to vector databases can reverse-engineer vectors to reconstruct the original plaintext documents. Encrypt vector tables at rest and restrict access permissions.

## 7. Performance Considerations

- **Recall vs. Latency Trade-off:**
  - Higher index search parameters (e.g. `efSearch = 64`) increase query recall (accuracy) but increase search latency and CPU cost.
  - Lower parameters speed up queries but miss relevant results. Profile and find the optimal balance.

## 8. Scalability Considerations

- **Horizontal Partitioning (Sharding):** Shard vector databases using document metadata keys (like `tenant_id`) to ensure similarity queries execute within local, isolated shards, avoiding expensive scatter-gather searches across the network.

## 9. How Major Companies Implement It

- **Pinterest:** Deploys HNSW indexes across millions of images, enabling real-time visual recommendation search results with sub-50ms latencies.
- **Spotify:** Utilizes vector databases to index music tracks and user listening profiles, executing similarity matching to generate personalized playlists.

## 10. Decision Checklist (Vector Index Selection)

Select the vector index type:

- Use **HNSW (Hierarchical Navigable Small World)** when:
  - Sub-10ms search latency is a strict SLA requirement.
  - High search recall (accuracy) is critical.
  - The server has sufficient budget to keep the entire index cached in RAM.
- Use **IVF (Inverted File Index) with Quantization** when:
  - Data volume is massive (>10M vectors) and RAM budget is limited.
  - Minor search recall loss (1-2%) is acceptable.
  - Index build times must be kept low.
- Never use **Vector Indexes** for:
  - Exact match scalar lookups (e.g. searching by User ID or status). Use B-Trees.

## 11. AI Coding-Agent Implementation Guidelines

- Always match the database distance metric to the embedding model specification.
- Never write database configuration files that allow vector indexes to exceed host RAM limits.
- Always configure HNSW index parameters (`M`, `efConstruction`) explicitly in DDL statements.
- Never recommend running synchronous vector index generation on tables containing over 100,000 rows.
- Always recommend sharding vector tables by tenant ID in multi-tenant SaaS environments.

## 12. Reusable Checklist

- [ ] Database distance metric (Cosine/Dot Product/L2) matched to the embedding model
- [ ] Entire vector index sized to fit completely within database RAM
- [ ] HNSW parameters (`M`, `efConstruction`) tuned based on recall requirements
- [ ] Vector quantization (SQ8/PQ) active if memory optimization is required
- [ ] Vector table sharded by metadata partition key (e.g., `tenant_id`)
- [ ] Index build execution runs asynchronously in the background
- [ ] Vector search query latency and recall monitored under production-scale loads
- [ ] Access controls and encryption active on tables containing vector arrays
