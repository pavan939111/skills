# Logging

## 1. Definition & Core Concepts

Logging is the active record of events, state transitions, and anomalies generated by an application during execution. It serves as the primary post-hoc diagnostic tool for operators and developers.

Core pieces:
- **Structured Logging:** Formatting log messages as machine-readable data (typically JSON) rather than raw text strings. This enables programmatic parsing, querying, and analysis.
- **Log Levels:** Standard classifications of message severity:
  - `DEBUG`: Verbose diagnostic info for development/troubleshooting.
  - `INFO`: Normal operational milestones (e.g., service startup, batch job completion).
  - `WARN`: Non-blocking anomalies that require attention but don't halt processing (e.g., high latency, retried requests).
  - `ERROR`: Event failures affecting a specific request/transaction (e.g., database connection failure, external API timeout).
  - `FATAL`/`CRITICAL`: Severe errors that crash the process or leave the application in an unusable state.
- **Log Context/Metadata:** Enrichment data included in every log entry, such as `trace_id`, `span_id`, `user_id`, `environment`, and `version`.
- **Log Sinks:** The destination where logs are sent (typically standard output/stdout in cloud-native applications, or local files in legacy servers).

## 2. Why It Exists

In distributed production environments, debuggers cannot be attached to live containers, and individual servers cannot be logged into manually. Centralized logs allow developers to reconstruct the sequence of events leading to a failure, audit system access, trace transaction lifetimes across services, and monitor system health.

## 3. What Breaks in Production Without It

- **Blind Troubleshooting:** When an error occurs, developers must guess the root cause because there are no records of the inputs, state, or stack trace.
- **Leaked Secrets:** Without standardized, intercepted logging, developers inevitably log raw objects that contain passwords, credit card numbers, or auth tokens.
- **Disk Exhaustion:** Directing logs to local files without rotation rules fills server disk space, crashing the entire operating system.
- **Performance Drag:** Synchronous logging (blocking execution to write to disk or network) introduces massive latency spikes in high-throughput hot paths.
- **Trace Discontinuity:** Without propagating trace IDs, it is impossible to connect logs from a single client request across multiple microservices.

## 4. Best Practices

- **Log to stdout/stderr:** In containerized/cloud environments, let the application write exclusively to standard streams. The platform orchestrator (Docker, Kubernetes) or a sidecar daemon (Fluentbit, Logstash) should handle aggregation, rotation, and transport.
- **Enforce Structured JSON:** Output all production logs as single-line JSON objects.
- **Correlate with Trace IDs:** Inject a unique `trace_id` at the entry point of every transaction/request and propagate it to every log statement, thread, and downstream network call.
- **Redact Sensitive Data:** Implement global interceptors or filters in the logging library to scrub passwords, credit card numbers, PII (emails, names), and API keys.
- **Use Log Levels Correctly:** Prevent log pollution by using the appropriate level. Avoid logging normal database queries at `INFO` level.
- **Validate Log Formatting at Boot:** Ensure the logger is initialized correctly before any other module executes, failing fast if the configuration is invalid.

## 5. Common Mistakes / Anti-Patterns

- **Standard Print Statements:** Using `console.log`, `print()`, or `System.out.println` which lacks log levels, timestamps, structured metadata, and asynchronous writing.
- **Log and Throw:** Logging an error and then throwing the exception up the call stack. This results in the same error being logged multiple times, inflating log volume and confusing debuggers.
- **String Interpolation:** Using string interpolation (e.g., `logger.info(f"User {id} logged in")`) which prevents indexers from caching the log template and opens vectors for log injection. Use parameterized logs (e.g., `logger.info("User logged in", {"user_id": id})`).
- **Synchronous Disk Writing:** Configuring log libraries to write directly to files synchronously, blocking the main execution thread.
- **No Rate Limiting on Errors:** Flooding log systems with thousands of identical error logs during a database outage, leading to log collector rate-limiting or heavy financial charges.

## 6. Security Considerations

- **Log Injection:** Untrusted user input printed to logs can manipulate log records (e.g., injecting newline characters and fake log entries) or trigger code execution (like the Log4j vulnerability). Sanitize or structure all user-controlled data.
- **Data Protection Compliance (GDPR/HIPAA):** Ensure no Protected Health Information (PHI) or Personally Identifiable Information (PII) is logged. If PII is necessary for operations, encrypt it or hash it securely.
- **Access Control:** Restrict access to log management platforms (Datadog, Splunk, Elastic) using Role-Based Access Control (RBAC) and Single Sign-On (SSO).

## 7. Performance Considerations

- **Asynchronous Appenders:** Use memory-buffered, asynchronous appenders for logging. The main execution thread writes to an in-memory queue, and a background thread flushes it.
- **Log Level Checks:** Wrap expensive log computations (e.g., serializing a complex object) in conditional checks: `if (logger.isDebugEnabled()) { logger.debug(...) }`.
- **JSON Serialization Cost:** JSON serialization has a CPU cost. Avoid logging massive data payloads (e.g., full request/response bodies) on high-volume routes.

## 8. Scalability Considerations

- **Log Volume Controls:** Implement dynamic log level switching at runtime (e.g., via an API endpoint or feature flag) to temporarily enable `DEBUG` logs during incidents without redeploying.
- **Log Sampling:** In high-volume environments (e.g., ad tech, IoT), sample logs on success paths (log 1% of successful transactions, but 100% of failures) to manage bandwidth and storage costs.

## 9. How Major Companies Implement It

- **Stripe:** Binds request metadata (merchant ID, API key ID, client IP) to a thread-local or context-propagated logger at the API boundary, guaranteeing every downstream log automatically includes complete diagnostic context.
- **Netflix:** Utilizes structured logs parsed by Logstash sidecars, shipping them to Elasticsearch clusters. They utilize dynamic configuration to change log levels across running containers without restart.
- **Uber:** Developed `zap`, a highly optimized, structured Go logging library designed specifically to minimize heap allocations and CPU overhead in high-throughput microservices.

## 10. Decision Checklist

- Use **Structured JSON Logging** when: The app runs in production, microservices need correlation, or logs are aggregated by systems like Datadog, Splunk, or Elasticsearch.
- Use **Plain Text Logging** when: Running local development environments (for human readability, often colorized).
- Enable **Asynchronous Logging** when: The service is high-throughput or latency-sensitive.

## 11. AI Coding-Agent Implementation Guidelines

- Always initialize a dedicated, structured logging library (e.g., Winston, Zap, Logback, Structlog) at startup — never use raw console/print statements.
- Always output JSON formatted logs in production environments.
- Always extract and include the `trace_id` (and `span_id` if available) in all request-scoped log statements.
- Never write credentials, API keys, tokens, or PII (e.g., passwords, SSNs, credit cards) directly to logs. Use a utility to redact or mask fields.
- Never log an exception and throw it immediately after. Log only at the boundary where the exception is handled/swallowed, or propagate it up.
- Always parameterize logs rather than using string interpolation for variables.
- Always set up log rotation and retention policies if writing logs to local files is unavoidable.

## 12. Reusable Checklist

- [ ] All print/console statements replaced with structured logger calls
- [ ] Production logs output as structured JSON
- [ ] No secrets, tokens, or PII logged in plaintext
- [ ] Trace and Span IDs automatically propagated to logs
- [ ] Log levels used appropriately (`DEBUG` for verbose, `WARN` for retryable anomalies, `ERROR` for request failures)
- [ ] Parameterized logging used instead of raw string interpolation
- [ ] Logging is asynchronous/non-blocking in high-throughput code paths
- [ ] Logger configuration validated at startup (fails fast if invalid)
