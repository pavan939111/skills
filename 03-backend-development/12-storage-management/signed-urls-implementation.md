# Signed URLs

## 1. Definition & Core Concepts
Signed URLs are time-limited, secure URL endpoints generated by the backend that grant temporary read or write access to private objects in storage buckets.

## 2. Why It Exists / What Problem It Solves
Private storage buckets cannot be accessed by clients. Signed URLs let the backend delegate temporary access directly to users, avoiding routing file byte downloads through backend memory.

## 3. What Breaks in Production Without It
- **Data Exposures:** Buckets are made public to allow client downloads, exposing files to unauthorized scans.
- **Server Thread Exhaustion:** Backend processes spend CPU and memory downloading and streaming large files to clients.

## 4. Best Practices
- **Configure Short Expirations:** Set signed URL lifespans under 15-30 minutes.
- **Lock HTTP Verbs:** Generate signed URLs restricted to specific methods (GET for downloads, PUT for uploads).
- **Match IP Restrictions:** Restrict signed URL execution coordinates to the requester's IP when security demands it.

## 5. Common Mistakes / Anti-Patterns
- **Generating long-lived URLs:** Setting URL lifespans to days or weeks, making links shareable.
- **Executing remote API calls during URL generation:** Generating URLs via remote cloud calls. Use local cryptographic libraries to sign keys.

## 6. Security Considerations
- **Signature Validations:** Use AWS/GCP SDK signature mechanics to ensure links cannot be modified in transit.

## 7. Performance Considerations
- **Local Signing:** Generate signatures offline in code using the SDK's local cryptographic key configurations.

## 8. Scalability Considerations
- **Delegated Client Uploads:** Use signed PUT URLs to let clients upload files directly to S3.

## 9. How Major Companies Implement It
- **Slack:** Generates short-lived signed URLs for all user-shared file attachments, securing files.

## 10. Decision Checklist (Access Control)
- Use **Signed URLs** when:
  - Serving private user documents, billing statements, or allowing direct uploads to S3.
- Use **Public CDN Links** when:
  - Serving generic static assets (landing page images, logos, styles).

## 11. AI Coding-Agent Guidelines
- Write signing methods that use local keys to generate expiration timestamps and cryptographic hashes.

## 12. Reusable Checklist
- [ ] Signed URL lifespans set to expire under 30 minutes
- [ ] URL generation executed locally in memory using SDK cryptographies
- [ ] Specific HTTP methods (GET/PUT) locked in signed parameters
- [ ] Private file downloads delegate to signed URLs (buckets remain private)
- [ ] CORS policies on buckets configured to restrict PUT uploads
- [ ] Verification tests check URL expiration timeouts
